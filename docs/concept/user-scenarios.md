## Посетитель (B2C)

### B2C-1: Уточнение запроса

**Актор:** посетитель сайта (B2C)

**Предусловия:** виджет открыт на llmstart.ru или пользователь в Telegram-боте.

**Основной поток:**
1. Пользователь задаёт вопрос о курсах или обучении.
2. Агент уточняет потребность (уровень, цель, формат).
3. Агент определяет сегмент **B2C** и фильтрует RAG по `data/b2c/`.

**Результат:** сегмент зафиксирован в сессии, агент готов к подбору продукта.

**Данные:** Session (segment=B2C, channel, message history).

---

### B2C-2: Подбор продукта

**Актор:** посетитель (B2C)

**Предусловия:** сегмент B2C определён (B2C-1).

**Основной поток:**
1. Агент вызывает `list_b2c_products` и/или `search_knowledge_base` (фильтр b2c).
2. Агент рекомендует 1–2 продукта из каталога с обоснованием.
3. В веб-виджете пользователь видит reasoning и вызванные tools.

**Результат:** пользователь понимает, какой курс подходит.

**Данные:** Product (id, name), RAG chunks из `data/b2c/`.

---

### B2C-3: Мок-оплата

**Актор:** посетитель (B2C)

**Предусловия:** продукт выбран или пользователь явно готов купить.

**Основной поток:**
1. Пользователь просит оплатить или агент предлагает покупку.
2. Агент вызывает `create_payment_link` — генерируется **мок-ссылка**.
3. Агент отправляет ссылку пользователю (формат зависит от channel).

**Результат:** мок-ссылка на оплату в сессии.

**Данные:** Payment (mock_link, product_id, status=pending).

---

### B2C-4: Подтверждение оплаты

**Актор:** посетитель (B2C)

**Предусловия:** мок-ссылка выдана (B2C-3).

**Основной поток:**
1. Пользователь пишет «оплатил» (или аналог).
2. Агент вызывает `confirm_payment` — статус меняется на **paid** (мок).
3. Агент благодарит и переходит к сбору контакта.

**Результат:** оплата подтверждена в рамках сессии (без реальной платёжки).

**Данные:** Payment (status=paid).

---

### B2C-5: Фиксация лида

**Актор:** посетитель (B2C)

**Предусловия:** оплата подтверждена (B2C-4) или пользователь готов оставить контакт.

**Основной поток:**
1. Агент запрашивает контакт (email, Telegram, имя).
2. Пользователь передаёт данные.
3. Агент вызывает `save_lead` — запись добавляется в `data/leads.txt`.

**Результат:** лид сохранён для команды LLMStart.

**Данные:** Lead (contact, product, segment=B2C, timestamp).

---

### B2C-6: Переход в Telegram

**Актор:** посетитель (B2C)

**Предусловия:** диалог начат в веб-виджете.

**Основной поток:**
1. В виджете пользователь выбирает «Перейти в Telegram».
2. Система передаёт идентификатор сессии / deep link в бот.
3. Диалог продолжается в Telegram с тем же контекстом (in-memory сессия backend).

**Результат:** channel меняется на `telegram`, история сохранена в рамках сессии.

**Данные:** Session (channel=telegram, session_id).

## Посетитель (B2B)

### B2B-1: Уточнение запроса

**Актор:** посетитель (B2B)

**Основной поток:**
1. Пользователь спрашивает о корпоративном обучении или разработке под заказ.
2. Агент уточняет масштаб, команду, задачу.
3. Агент определяет сегмент **B2B** и фильтрует RAG по `data/b2b/`.

**Результат:** сегмент B2B зафиксирован.

---

### B2B-2: Консультация по B2B

**Актор:** посетитель (B2B)

**Предусловия:** сегмент B2B (B2B-1).

**Основной поток:**
1. Агент вызывает `search_knowledge_base` (фильтр b2b).
2. Агент отвечает по материалам корпоративного обучения / разработки.
3. При off-topic запросе — ответ «не знаю», без выдумывания.

**Результат:** пользователь получил релевантную B2B-информацию.

---

### B2B-3: Фиксация лида

**Актор:** посетитель (B2B)

**Основной поток:**
1. Агент предлагает оставить контакт для связи с командой.
2. Пользователь передаёт данные компании и контактное лицо.
3. Агент вызывает `save_lead` → `data/leads.txt`.

**Результат:** B2B-лид сохранён.

---

## Администратор / преподаватель

### ADM-1: Просмотр traces

**Актор:** администратор / преподаватель

**Основной поток:**
1. Запускается тестовый диалог через виджет или бот.
2. В Langfuse (local) открывается trace: шаги ReAct, tools, latency, tokens.
3. Преподаватель анализирует качество и объясняет студентам.

**Результат:** observability подтверждена, trace доступен для разбора.

---

### ADM-2: Демо на занятии

**Актор:** администратор / преподаватель

**Основной поток:**
1. На проекторе открывается веб-виджет.
2. Задаётся типовой вопрос (подбор курса).
3. Студенты видят SSE-стриминг, reasoning, вызовы tools в реальном времени.

**Результат:** демонстрация «инженерной» работы агента.

---

## Студент курса

### STU-1: Изучение архитектуры

**Актор:** студент курса

**Основной поток:**
1. Студент поднимает стек локально (`make dev`).
2. Отправляет запрос через виджет или Telegram.
3. Прослеживает путь: клиент → backend API → ReAct → tools → RAG → ответ.
4. Смотрит trace в Langfuse и код backend/bot/frontend.

**Результат:** понимание архитектуры «один мозг — много каналов».

---

## Off-topic и нестандартные запросы

**Поведение (все роли):** если запрос вне каталога и базы знаний — агент отвечает **«не знаю»**, не генерирует вымышленные услуги.

**Пример:** «Сделайте HR-бота под нашу компанию» → «не знаю» / предложение оставить контакт для команды (B2B-3), без обещания услуги.
