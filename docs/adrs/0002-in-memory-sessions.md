# ADR 0002 — In-memory сессии без Postgres в MVP

> **Статус:** ✅ Accepted
> **Дата:** 2026-06-21
> **Автор:** LLMStart Agent team
> **Область:** продукт LLMStart Agent
> **Supersedes:** —

---

## 1. Контекст

### 1.1 Текущая ситуация

Диалоги агента требуют истории сообщений и контекста сессии (channel, segment). В MVP БД не используется.

### 1.2 Что породило необходимость решения

Нужно хранить состояние диалога между запросами без поднятия Postgres.

### 1.3 Силы, действующие на решение

| Сила | Вес | Следствие |
|------|-----|-----------|
| YAGNI | Высокий | Postgres — в roadmap |
| Фокус курса | Высокий | Архитектура агента, не БД |
| Простота деплоя | Средний | Меньше инфраструктуры |

---

## 2. Рассмотренные варианты

### Вариант A. In-memory dict в backend

**Плюсы:** простота, нулевая инфраструктура

**Минусы:** сброс при рестарте, нет масштабирования на несколько инстансов

**Вердикт:** принят для MVP

### Вариант B. PostgreSQL с первого дня

**Минусы:** scope MVP, миграции, Alembic — не цель курса сейчас

**Вердикт:** отклонён для MVP, планируется позже

---

## 3. Решение

**Принимаем: in-memory хранилище сессий в backend.**

- Session: id, messages, channel, segment, payment state
- При рестарте backend — сессии теряются (допустимо для учебного стенда)
- Переход виджет → Telegram — в рамках одной session_id

---

## 4. Последствия

### 4.2 Негативные (и митигация)

| Риск | Митигация |
|------|----------|
| Потеря диалога при рестарте | Документировано; Postgres в roadmap |
| Один инстанс backend | Достаточно для MVP и локальной разработки |
